# predictionmarkets_rec 推荐服务接口文档

推荐服务对外只暴露一个接口：`PredictionMarketsRecommend.recommend`。基于 brpc + Protobuf 实现，本文档只描述 HTTP + JSON 调用方式。

- 协议定义：`proto/recommend.proto`（package `predictionmarkets_rec`）
- 服务 / 方法：`PredictionMarketsRecommend` / `recommend`
- 调用方式：`POST`，Content-Type `application/json`
- URL：`http://<host>:15555/PredictionMarketsRecommend/recommend`
- 端口：`15555`（`conf/config.conf`、`conf/config.prod.conf` 一致，可配 `server.port`）

---

## 一、请求参数（RecRequest）

| 顺序 | 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| 1 | `user_id` | string | 如果存在则必填 | 登录用户标识，member uuid |
| 2 | `device_id` | string | 没有 user_id 时必填 | 设备标识；user_id 为空时用它作锚点（anchor） |
| 3 | `timestamp` | int64 | 是 | 请求时间，Unix 秒；用于内容时效判定，传毫秒或 0 会导致时效规则错乱 |
| 4 | `request_id` | string | 建议 | 请求唯一标识，透传回响应，用于链路追踪与日志核对 |
| 5 | `session_id` | string | 建议 | 会话标识，用于跨刷打散 |
| 6 | `session_refresh_num` | int32 | 建议 | 当前会话第几刷，0 表示首屏；影响时效规则与跨刷打散 |
| 7 | `topk` | int32 | 是 | 期望返回的条数上限；池内候选不足时可能少于该值 |
| 8 | `is_paid_user` | bool | 否 | 是否付费用户；true 走全量池，false 或缺省走免费池（非付费内容） |

### 锚点（anchor）与分组
- `anchor_id = user_id 非空 ? user_id : device_id`；两者都为空 → 返回 400。
- 实验分组：`group = MD5(anchor_id + "_predictionmarkets") % 10`（group_0 ~ group_9）。
- 黑名单 / 曝光历史 / 关注 / 推荐历史等按 `anchor_id` 查找。

## 二、响应结构（RecResponse）

```json
{
  "code": 200,
  "message": "success",
  "request_id": "req-123",
  "timestamp": 1752380000,
  "data": {
    "current_refresh_num": 0,
    "feed_list": [
      { "item_id": "69b0d6052d1d5b00013a1aa7", "score": 0.0731 },
      { "item_id": "69b0d6052d1d5b00013a1aa8", "score": 0.0524 }
    ],
    "ext_info": {
      "strategy": "base"
    }
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int32 | 返回码，见第三节 |
| `message` | string | 返回码对应文案 |
| `request_id` | string | 透传自请求 |
| `timestamp` | int64 | 透传自请求 |
| `data.current_refresh_num` | int32 | 回显请求的 `session_refresh_num` |
| `data.feed_list` | array | 推荐结果列表，按最终排序（越靠前越优先），长度 ≤ `topk` |
| `data.feed_list[].item_id` | string | 内容 id（Ghost post id） |
| `data.feed_list[].score` | float | 排序分 |
| `data.feed_list[].callback_feature` | string | 预留字段，逐 item 的线上特征经此回传；当前留空，空值时 JSON 不返回（proto3 行为），有值后返回 |
| `data.ext_info.strategy` | string | 命中的策略标识 |

---

## 三、返回码

| code | message | 含义 |
|------|---------|------|
| `200` | `success` | 成功 |
| `400` | `missing user_id and device_id` | `user_id` 与 `device_id` 均为空，无法锚定 |
| `500` | `item pool not ready` | item 池尚未加载完成（服务刚启动、上游未就绪） |
| `500` | `internal error` | 处理过程中抛出异常（详见服务端日志的 `request_id`） |

---

## 四、当前阶段（阶段一 · bandit 冷启动）说明

以下为当前实现的临时/占位行为，随迭代会变化，接入方需知悉：

- `score`：阶段一无特征与排序模型，排序分由汤普森采样得到（Beta 后验采样的预估点击率），是 [0,1] 的概率值，不是排序模型分。阶段二引入模型后语义会变。
- `strategy`：当前恒为 "base"（占位，尚未按实际命中策略区分）。
- `feed_list` 少于 `topk`：正常现象（内容池小、已曝光软降权等），不代表错误。
- `is_paid_user`：需调用方填充，缺省 false → 免费池。
- `callback_feature`：预留字段，逐 item 回传该内容的线上特征；当前恒为空字符串，空值时不出现在 JSON 响应里，有值后返回。

## 五、调用示例

已登录用户（填 `user_id`）：

```bash
curl -X POST "http://127.0.0.1:15555/PredictionMarketsRecommend/recommend" \
     -H "Content-Type: application/json" \
     -d '{
           "user_id": "2f6d0e1a-9c3b-4a7e-b8d5-1c4f7a0e9b23",
           "timestamp": 1752380000,
           "request_id": "req-123",
           "session_id": "sess-abc",
           "session_refresh_num": 0,
           "topk": 10,
           "is_paid_user": false
         }'
```

未登录（仅设备标识，`user_id` 留空、填 `device_id`）：

```bash
curl -X POST "http://127.0.0.1:15555/PredictionMarketsRecommend/recommend" \
     -H "Content-Type: application/json" \
     -d '{"device_id":"dev-xyz","timestamp":1752380000,"request_id":"req-124","topk":10}'
```

## 六、id 空间提醒

`user_id` 用的是 Ghost member（受众/读者）的 uuid，与 Ghost author（作者/内容生产者）的 id 是两套独立空间。上游写入 redis 的黑名单 / 曝光历史 / 关注等 key，必须与请求里的 `user_id`（member uuid）保持同一口径，否则推荐侧按 member uuid 读、上游按别的 id 写会静默失效。
